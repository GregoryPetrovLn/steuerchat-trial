# Architecture Plan

## Overview

A real-time streaming chat system: NestJS backend streams a ~500-word text over socket.io at ~5 words/sec; native SwiftUI iOS app displays chunks as they arrive, supports cancel and reconnect/resume mid-stream.

## Key Design Decisions

### 1. Session Identity: Client-Generated `messageId`

Each message gets a UUID generated by the client. This `messageId` is sent with every event (send, resume, cancel). The server keys all stream state by `messageId`. socket.io's `socket.id` changes on reconnect, so we never rely on it for identity.

### 2. Server Behavior During Disconnect: **Buffer (Option A)**

We choose **option (a): keep advancing the stream into an in-memory buffer** during disconnect.

**Rationale:** This matches real-world LLM behavior where the model keeps generating tokens regardless of client state. It also satisfies the requirement that "missing chunks appear quickly after reconnect" — the buffer is already filled, so we flush it instantly. The alternative (pause) would mean the user waits the same total time, which feels broken from a UX perspective.

The buffer is bounded by the text length (~500 words), so memory is not a concern.

### 3. Cancel vs. Disconnect

| Event | Trigger | Server Action |
|-------|---------|---------------|
| **Cancel** | Client sends `cancel` event with `messageId` | Server destroys the stream state immediately. Client shows partial response as final. |
| **Disconnect** | Transport-level (socket.io `disconnect` event) | Server keeps streaming into buffer. State lives for TTL (60s). After TTL, garbage-collected. |

### 4. Catch-Up After Reconnect

When client sends `resume` with `{ messageId, lastWordIndex }`:
1. Server finds the active session by `messageId`.
2. Sends all words from `lastWordIndex+1` to current position as a single `catch-up` event.
3. Resumes normal streaming at ~5 wps from current position.

### 5. Storage

In-memory `Map<string, StreamSession>`. No persistence. Server restart loses all state. Documented as a known limitation.

## Message Protocol

| Event Name | Direction | Payload | Description |
|------------|-----------|---------|-------------|
| `send-message` | Client → Server | `{ messageId: string, text: string }` | User sends a chat message. Server starts streaming response. |
| `stream-chunk` | Server → Client | `{ messageId: string, word: string, index: number }` | Single word of the streamed response. |
| `stream-end` | Server → Client | `{ messageId: string, totalWords: number }` | Stream completed normally. |
| `cancel` | Client → Server | `{ messageId: string }` | Client cancels active stream. |
| `stream-cancelled` | Server → Client | `{ messageId: string, lastIndex: number }` | Confirms cancellation. |
| `resume` | Client → Server | `{ messageId: string, lastWordIndex: number }` | Client reconnected, requests catch-up. |
| `catch-up` | Server → Client | `{ messageId: string, words: Array<{word: string, index: number}> }` | Batch of missed words sent rapidly. |
| `error` | Server → Client | `{ messageId?: string, message: string }` | Error notification. |

## StreamSession State Machine

```
                    send-message
                        │
                        ▼
                   ┌─────────┐
                   │STREAMING │◄──── resume (client reconnected)
                   └────┬─────┘       │
                        │             │
              ┌─────────┼─────────────┘
              │         │
         disconnect     │    cancel
              │         │      │
              ▼         │      ▼
        ┌──────────┐    │  ┌───────────┐
        │BUFFERING │    │  │CANCELLED  │──► (state destroyed)
        └────┬─────┘    │  └───────────┘
             │          │
        reconnect       │  stream complete
             │          │      │
             ▼          │      ▼
        ┌──────────┐    │  ┌───────────┐
        │CATCH-UP  │────┘  │COMPLETED  │──► (state destroyed after TTL)
        └──────────┘       └───────────┘
```

### States

| State | Description |
|-------|-------------|
| `STREAMING` | Active client connected, words emitting at ~5 wps |
| `BUFFERING` | Client disconnected, words still accumulating in buffer |
| `CANCELLED` | Client explicitly cancelled, state about to be destroyed |
| `COMPLETED` | All words sent, kept briefly for late resume attempts |

### Transitions

| From | Event | To |
|------|-------|----|
| (none) | `send-message` | STREAMING |
| STREAMING | `disconnect` | BUFFERING |
| STREAMING | `cancel` | CANCELLED → destroyed |
| STREAMING | stream complete | COMPLETED |
| BUFFERING | `resume` | STREAMING (after catch-up flush) |
| BUFFERING | TTL expires (60s) | destroyed |
| BUFFERING | stream complete | COMPLETED |
| COMPLETED | `resume` | send catch-up → COMPLETED |
| COMPLETED | TTL expires (30s) | destroyed |

### Key Implementation Details

- **Tick loop**: `setInterval` at 200ms (5 words/sec). Each tick: if state is STREAMING and client connected, emit `stream-chunk`. If BUFFERING, push word to buffer array.
- **GC timer**: Runs every 30s, removes sessions past TTL.
- **Socket mapping**: Maintain a `Map<messageId, socketId>` to know which socket to emit to. Updated on resume.

```
    ├── ChatView.swift              # Main chat screen
    ├── MessageBubble.swift         # Single message bubble
    └── ConnectionBanner.swift      # Status bar for connection state
```

### Library Choice: socket.io-client-swift

**Rationale:** The backend uses socket.io, so using the official Swift client avoids hand-rolling the socket.io protocol (Engine.IO framing, namespace multiplexing, ack IDs). Starscream is raw WebSocket only — we'd have to implement socket.io on top. The official client handles reconnect with exponential backoff out of the box.

### Reconnect Flow (iOS)

1. socket.io-client-swift detects disconnect → `ChatViewModel.connectionState = .reconnecting`
2. Library auto-reconnects with backoff.
3. On `connect`, if there's an active `messageId` with `lastWordIndex`, emit `resume { messageId, lastWordIndex }`.
4. Receive `catch-up` → append words to displayed text.
5. Resume receiving `stream-chunk` events normally.
6. `connectionState = .connected`.
