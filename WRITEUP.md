# Технический разбор

## Обзор

Это прототип стримингового чата с бэкендом на NestJS и нативным iOS-клиентом на SwiftUI. Сервер стримит предопределённый текст (~500 слов, отрывок из "Сердца-обличителя" Эдгара Аллана По) через socket.io со скоростью 5 слов в секунду. Клиент отображает слова по мере поступления, поддерживает отмену посреди стрима и прозрачно возобновляет передачу после временных разрывов сети.

---

## Жизненный цикл WebSocket

Бэкенд использует декоратор `@WebSocketGateway` из NestJS для создания socket.io-сервера (`chat.gateway.ts:20`). Gateway реализует интерфейсы `OnGatewayConnection` и `OnGatewayDisconnect` для хуков жизненного цикла соединения.

При подключении клиента `handleConnection` логирует новый socket ID (`chat.gateway.ts:39-41`). На стороне iOS `SocketService` создаёт `SocketManager`, указывающий на `localhost:3000`, с параметрами `forceWebsockets(true)` и включённым автореконнектом (`SocketService.swift:31-39`). Приложение вызывает `connect()` при появлении `ChatView` (`ChatView.swift:43`).

Gateway хранит две карты для связки сокет/сообщение (`chat.gateway.ts:32-33`):
- `messageToSocket: Map<string, string>` -- messageId к текущему socketId
- `socketToMessages: Map<string, Set<string>>` -- socketId ко всем его messageId

Эта двунаправленная связка критически важна: при отключении сокета сервер может найти все его активные стримы и поставить их на паузу.

---

## Реализация стриминга

Стриминг использует тик-подход: `setInterval` с интервалом 200мс (5 слов/сек) в `StreamSession.start()` (`stream-session.ts:47-49`). Каждый тик вызывает `tick()` (`stream-session.ts:53-79`), который:

1. Проверяет, исчерпан ли корпус -- если да, вызывает `complete()`.
2. Читает текущее слово из `CORPUS_WORDS` по индексу `currentIndex`.
3. Если состояние `STREAMING` и callback существует -- эмитит `stream-chunk` с `{ messageId, word, index }`.
4. Если состояние `BUFFERING` -- добавляет `{ word, index }` во внутренний буфер.
5. Инкрементирует `currentIndex`.

Emit callback -- это замыкание, создаваемое в `handleSendMessage` (`chat.gateway.ts:103-109`). Оно ищет текущий socketId для данного messageId и эмитит именно в этот сокет через `server.to(socketId).emit()`. Благодаря этой косвенности callback всегда попадает в правильный сокет, даже после реконнекта.

**Почему такой подход:** таймер с фиксированным интервалом прост, детерминирован и удобен для тестирования. Он имитирует стриминг токенов от LLM, где токены приходят с примерно постоянной скоростью. Индекс слова в каждом чанке позволяет клиенту обнаруживать пропуски, а серверу -- вычислять диапазоны для догоняющей отправки.

## Идентификация сессии

Клиент генерирует UUID для каждого сообщения (`ChatViewModel.swift:39`) и отправляет его как `messageId` вместе с событием `send-message`. Все последующие события (`cancel`, `resume`, `stream-chunk`, `catch-up`, `stream-end`) ссылаются на этот `messageId`.

**Почему не socket.id?** Socket.io назначает новый `socket.id` при каждом подключении. После обрыва сети и реконнекта клиент получает другой `socket.id`. Если бы сессии были привязаны к `socket.id`, сервер не смог бы связать новое соединение со старым стримом. Клиентский `messageId` стабилен между реконнектами.

Сервер хранит три карты:
- `SessionManager.sessions: Map<messageId, StreamSession>` (`session-manager.ts:12`)
- `ChatGateway.messageToSocket: Map<messageId, socketId>` (`chat.gateway.ts:32`)
- `ChatGateway.socketToMessages: Map<socketId, Set<messageId>>` (`chat.gateway.ts:33`)

## Поток отмены

1. Клиент вызывает `cancelStream(messageId:)` (`ChatViewModel.swift:56-58`), который эмитит `cancel { messageId }` через `SocketService.cancelStream` (`SocketService.swift:63-65`).
2. Серверный `handleCancel` (`chat.gateway.ts:114-140`) находит сессию и вызывает `session.cancel()` (`stream-session.ts:156-159`), который очищает интервальный таймер и устанавливает состояние `CANCELLED`.
3. Сервер эмитит `stream-cancelled { messageId, lastIndex }` обратно клиенту.
4. Сервер удаляет сессию из карт привязки сокетов и из менеджера сессий.
5. Клиент получает `stream-cancelled`, вызывает `finaliseStream()` (`ChatViewModel.swift:142-150`), который устанавливает `isStreaming = false` и очищает состояние отслеживания.

Отмена -- это явное, чистое завершение. Сессия уничтожается немедленно -- без TTL, без буфера.

## Поток disconnect / reconnect / resume

Это ключевая функциональность. Далее -- пошаговый разбор сценария:

### 1. Стриминг в процессе

Пользователь отправил сообщение. `StreamSession` находится в состоянии `STREAMING`, таймер с интервалом 200мс работает, слова эмитятся в подключённый сокет.

### 2. Обрыв сети

Транспортный уровень падает (режим полёта, переключение Wi-Fi и т.д.). Socket.io на сервере обнаруживает отключение и вызывает `handleDisconnect` (`chat.gateway.ts:43-61`).

### 3. Сервер ставит сессию на паузу

`handleDisconnect` перебирает все messageId, привязанные к отключённому сокету (`chat.gateway.ts:46-58`). Для каждого вызывает `session.pause()` (`stream-session.ts:100-112`):
- Состояние меняется с `STREAMING` на `BUFFERING`
- `socketId` и `emitCb` устанавливаются в `null`
- Записывается `disconnectedAt` для отслеживания TTL
- **Таймер продолжает работать.** Последующие тики добавляют слова в `session.buffer` вместо эмита (`stream-session.ts:69-71`).

### 4. Клиент показывает состояние реконнекта

На стороне iOS срабатывает обработчик disconnect в `SocketService` (`SocketService.swift:86-91`). `ChatViewModel` устанавливает `connectionState = .reconnecting` (`ChatViewModel.swift:78-81`). В UI появляется `ConnectionBanner` (`ChatView.swift:10-12`).

### 5. Сеть восстановлена, socket.io переподключается

socket.io-client-swift имеет встроенный автореконнект с экспоненциальным backoff (`SocketService.swift:35-37`: `reconnectWait(1)`, `reconnectWaitMax(5)`). Когда сеть возвращается, библиотека устанавливает новое соединение. Новое соединение получает новый `socket.id`.

### 6. Клиент отправляет resume

Срабатывает обработчик connect в `SocketService` (`SocketService.swift:80-84`). `ChatViewModel.onConnect` проверяет наличие активного стрима (`ChatViewModel.swift:70`): если `currentMessageId` установлен и `isStreaming` равен true, эмитится `resume { messageId, lastWordIndex }` (`ChatViewModel.swift:71-74`). `lastWordIndex` хранит наибольший индекс слова, успешно полученный клиентом (`ChatViewModel.swift:135-136`).

### 7. Сервер обрабатывает resume

`handleResume` в gateway (`chat.gateway.ts:142-199`):
1. Валидирует payload (`chat.gateway.ts:148-160`).
2. Находит сессию по messageId (`chat.gateway.ts:162`).
3. Обновляет карты привязки сокетов на новый сокет (`chat.gateway.ts:180-181`).
4. Создаёт новый emit callback, привязанный к новому сокету (`chat.gateway.ts:183-188`).
5. Вызывает `session.resume(socketId, lastWordIndex, emitCallback)` (`chat.gateway.ts:190`).

Внутри `StreamSession.resume()` (`stream-session.ts:118-153`):
1. Устанавливает новые `socketId` и `emitCb`.
2. Сбрасывает `disconnectedAt`.
3. Строит массив догоняющих слов: итерирует от `lastWordIndex + 1` до `currentIndex - 1`, восстанавливая пропущенные слова напрямую из массива корпуса (`stream-session.ts:130-132`). Это надёжнее буфера, потому что использует первоисточник.
4. Очищает буфер (`stream-session.ts:135`).
5. Если стрим завершился во время отключения -- отправляет `stream-end` (`stream-session.ts:137-144`).
6. В противном случае устанавливает состояние обратно в `STREAMING` (`stream-session.ts:149`).

### 8. Доставка догоняющих данных

Обратно в `handleResume`: если есть догоняющие слова, сервер эмитит `catch-up { messageId, words }` (`chat.gateway.ts:192-194`). На клиенте `onCatchUp` итерирует массив слов и вызывает `appendWord` для каждого (`ChatViewModel.swift:95-100`). Метод `appendWord` содержит guard: `index > lastWordIndex` (`ChatViewModel.swift:135`), предотвращающий дубликаты.

### 9. Нормальный стриминг продолжается

Таймер по-прежнему работает. Теперь, когда состояние снова `STREAMING` и существует валидный `emitCb`, последующие тики эмитят `stream-chunk` в новый сокет штатным образом. Клиент получает их и добавляет слова. Стрим завершается, когда все слова корпуса отправлены.

## Сборка мусора

`SessionManager` запускает GC-таймер каждые 30 секунд (`session-manager.ts:16-18`). Удаляются:
- Сессии в состоянии `BUFFERING` старше 60 секунд (`session-manager.ts:50-52`)
- Сессии в состоянии `COMPLETED` старше 30 секунд (`session-manager.ts:54-57`)
- Сессии в состоянии `CANCELLED` -- немедленно (`session-manager.ts:59`)

Модуль реализует `OnModuleDestroy` для очистки всех сессий и таймеров при завершении работы (`session-manager.ts:72-83`).

## Компромиссы

**Хранение в памяти.** Всё состояние сессий живёт в объектах `Map` внутри Node-процесса. Перезапуск сервера теряет все активные стримы. Для прототипа это допустимо, но не для продакшена.

**Буфер vs. пауза.** Я выбрал продолжение работы таймера при отключении с буферизацией слов. Альтернатива -- остановка таймера -- означала бы, что клиент ждёт столько же реального времени независимо от длительности отключения. Буферизация имитирует реальное поведение LLM (модель не прекращает генерацию из-за обрыва у клиента) и обеспечивает мгновенный catch-up.

**Нет аутентификации.** Любой клиент может подключиться, отправить сообщение и возобновить любой messageId. В продакшене нужна валидация владельца messageId при resume.

**Нет горизонтального масштабирования.** Сессии привязаны к процессу. При нескольких инстансах за балансировщиком переподключающийся клиент может попасть на другой инстанс, где нет данных о сессии. Решение -- Redis-backed сессии или sticky sessions.

**TTL 60с для отключённых сессий.** Произвольный, но разумный выбор. Слишком маленький -- пользователь не успеет восстановиться после короткого туннеля или лифта. Слишком большой -- сервер накапливает устаревшие сессии.

**Один параллельный стрим на клиента.** Архитектура поддерживает несколько стримов на сокет (через множество `socketToMessages`), но iOS-клиент отслеживает только один `currentMessageId`.

## Что бы я улучшил при наличии большего времени

- **Redis-backed сессии** для горизонтального масштабирования между несколькими инстансами сервера.
- **Аутентификация** (JWT handshake middleware в socket.io) для валидации идентичности клиента и владения messageId.
- **Обработка backpressure** -- проверка `socket.conn.transport.writable` перед эмитом; пауза таймера, если клиент не успевает.
- **Rate limiting** для предотвращения злоупотреблений (макс. параллельных стримов, макс. сообщений в минуту).
- **Персистентная история чата** -- хранение завершённых сообщений в базе данных для перезагрузки прошлых диалогов.
- **E2E-тесты с реальным iOS-симулятором** (XCTest + локальный сервер) в дополнение к юнит-тестам.
- **Метрики и observability** -- Prometheus-счётчики для активных сессий, завершений стримов, реконнектов и размеров буфера; структурированное JSON-логирование.
- **Конфигурируемая скорость стриминга** -- принимать параметр `wordsPerSecond` вместо хардкода 200мс.
- **Несколько параллельных стримов на клиента** -- возможность отправить новое сообщение, пока предыдущее ещё стримится.
